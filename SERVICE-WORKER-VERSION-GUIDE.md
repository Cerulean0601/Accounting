# Service Worker 版本號更新指南

## 📌 什麼時候需要更新版本號？

### ✅ **必須更新版本號的情況**

#### 1. **修改了 Service Worker 的緩存策略**
當你改變了資源的緩存方式時，必須更新版本號以清除舊緩存。

**範例**：
```javascript
// 舊版：緩存優先
caches.match(request).then(response => response || fetch(request))

// 新版：網路優先（已改進）
fetch(request).then(...).catch(() => caches.match(request))
```
**原因**：舊緩存可能包含錯誤的資源，需要清除。

---

#### 2. **修改了預載入的資源列表（urlsToCache）**
當你改變了 `urlsToCache` 陣列時，應該更新版本號。

**範例**：
```javascript
// 舊版
const urlsToCache = ['/', '/manifest.json', '/icon-192.png'];

// 新版：新增了資源
const urlsToCache = ['/', '/manifest.json', '/icon-192.png', '/new-resource.js'];
```
**原因**：確保新資源被正確預載入，舊緩存被清除。

---

#### 3. **修復了緩存相關的 Bug**
當你修復了 Service Worker 中與緩存相關的問題時，必須更新版本號。

**範例場景**：
- 修復了緩存損壞導致的 loading 問題
- 修復了緩存更新邏輯錯誤
- 修復了離線模式下的緩存問題

**原因**：舊緩存可能包含有問題的資源，需要強制清除。

---

#### 4. **改變了緩存命名規則或結構**
當你改變了緩存的名稱格式或組織方式時。

**範例**：
```javascript
// 舊版
const CACHE_NAME = 'accounting-app-v1';

// 新版：改變了命名規則
const CACHE_NAME = `accounting-app-${CACHE_VERSION}`;
```

---

#### 5. **需要強制所有用戶清除緩存**
當你發現舊緩存導致問題，需要強制所有用戶清除時。

**範例場景**：
- 用戶回報緩存導致頁面無法載入
- 發現舊緩存包含安全問題
- 需要強制更新到新版本

---

### ❌ **不需要更新版本號的情況**

#### 1. **只修改了業務邏輯（不涉及緩存）**
如果只是修改了 `storeOfflineTransaction`、`syncOfflineTransactions` 等函數的內部邏輯，但不影響緩存策略。

**範例**：
```javascript
// 只修改了同步邏輯，不影響緩存
async function syncOfflineTransactions() {
  // 修改了這裡的邏輯，但緩存策略沒變
}
```

---

#### 2. **只修改了註解或代碼格式**
純粹的代碼整理，不影響功能。

---

#### 3. **只修改了錯誤訊息文字**
不影響緩存行為的修改。

---

## 🔄 版本號更新流程

### 步驟 1：決定版本號格式

**建議使用語義化版本**：
- `v1` → `v2` → `v3`（簡單遞增）
- 或 `v1.0` → `v1.1` → `v2.0`（語義化版本）

### 步驟 2：修改版本號

在 `public/sw.js` 中修改：

```javascript
// 舊版
const CACHE_VERSION = 'v2';

// 新版
const CACHE_VERSION = 'v3';  // ← 修改這裡
```

### 步驟 3：測試更新流程

1. **清除本地緩存**（開發測試）
   - 打開開發者工具
   - Application → Service Workers → Unregister
   - Application → Cache Storage → 刪除所有緩存

2. **重新載入頁面**
   - 應該會自動註冊新的 Service Worker
   - 舊緩存應該被自動清除
   - 新緩存應該被創建

3. **驗證功能**
   - 測試正常載入
   - 測試離線功能
   - 測試緩存更新

## 📊 版本號更新決策樹

```
需要更新 Service Worker？
    │
    ├─ 修改了緩存策略？ ──→ YES ──→ 更新版本號
    │
    ├─ 修改了 urlsToCache？ ──→ YES ──→ 更新版本號
    │
    ├─ 修復了緩存相關 Bug？ ──→ YES ──→ 更新版本號
    │
    ├─ 需要強制清除舊緩存？ ──→ YES ──→ 更新版本號
    │
    └─ 只修改業務邏輯？ ──→ NO ──→ 不需要更新
```

## 🎯 實際範例

### 範例 1：修復 loading 問題（需要更新）

**情況**：發現緩存導致 Cursor 內建瀏覽器卡在 loading

**操作**：
```javascript
// 修改前
const CACHE_VERSION = 'v1';

// 修改後
const CACHE_VERSION = 'v2';  // 更新版本號
```

**原因**：需要清除可能包含問題資源的舊緩存。

---

### 範例 2：改進緩存策略（需要更新）

**情況**：從「緩存優先」改為「網路優先」

**操作**：
```javascript
// 修改前
const CACHE_VERSION = 'v2';

// 修改後
const CACHE_VERSION = 'v3';  // 更新版本號
```

**原因**：舊緩存策略可能導致使用過期資源，需要清除。

---

### 範例 3：只修改同步邏輯（不需要更新）

**情況**：改進了 `syncOfflineTransactions` 的錯誤處理

**操作**：
```javascript
// 不需要修改版本號
const CACHE_VERSION = 'v2';  // 保持不變
```

**原因**：不影響緩存策略，舊緩存仍然有效。

---

## ⚠️ 注意事項

### 1. **版本號更新會清除所有舊緩存**
- 用戶需要重新下載資源
- 首次載入可能稍慢
- 但能確保使用最新、正確的資源

### 2. **不要頻繁更新版本號**
- 每次更新都會清除緩存
- 影響用戶體驗（需要重新下載）
- 只在必要時更新

### 3. **版本號應該與 Git 版本對應**
- 建議在 Git commit 中記錄版本號更新
- 方便追蹤和回滾

### 4. **生產環境部署前測試**
- 在開發環境測試版本更新流程
- 確認舊緩存被正確清除
- 確認新功能正常運作

---

## 📝 版本號更新檢查清單

更新版本號前，確認：

- [ ] 是否修改了緩存策略？
- [ ] 是否修改了 `urlsToCache`？
- [ ] 是否修復了緩存相關 Bug？
- [ ] 是否需要強制清除舊緩存？
- [ ] 是否在開發環境測試過？
- [ ] 是否更新了相關文檔？

**如果以上任一項為「是」，則需要更新版本號。**

---

## 🔍 如何檢查是否需要更新版本號？

### 方法 1：代碼比對
比較當前版本與上一版本的 `sw.js`：
- 緩存策略是否改變？
- `urlsToCache` 是否改變？
- 緩存相關邏輯是否改變？

### 方法 2：問題追蹤
- 是否有用戶回報緩存相關問題？
- 是否需要強制清除舊緩存？

### 方法 3：功能變更
- 新功能是否依賴新的緩存策略？
- 舊緩存是否會影響新功能？

---

## 💡 最佳實踐

1. **保守原則**：不確定時，更新版本號更安全
2. **記錄變更**：在 commit message 中說明版本號更新的原因
3. **測試優先**：更新後在開發環境充分測試
4. **監控效果**：部署後監控用戶體驗，確認更新成功
